# 앱인토스 비게임 미니앱 — AI 작업 워크플로우 (BASE)

> **목적**: 앱인토스(비게임) 미니앱 프로젝트에서 AI가 항상 참조해야 할 **불변 규칙**과 **단계별 워크플로우**.
> **사용법**: 이 파일은 베이스. 새 프로젝트마다 맨 아래 "프로젝트별 추가 섹션"에 프로젝트 고유 내용을 덧붙여 사용한다.
> **Source of Truth**: https://developers-apps-in-toss.toss.im/ (의심스러우면 항상 공식 문서 재확인)

---

## 0. AI 작업 철칙 (모든 단계에 우선 적용)

1. **검수 영향 분석 우선**: 코드/설정을 만지기 전, 해당 변경이 아래 §2 검수 항목에 영향을 주는지 먼저 판단하고 사용자에게 알린다.
2. **모르면 묻기**: 토스 SDK·콘솔 동작이 불확실하면 추측 금지. 공식 문서를 WebFetch로 확인하거나 사용자에게 질문.
3. **수정 전 계획 제시**: 코드 작성 전 마크다운 계획 → 사용자 동의 → 실행 (사용자 글로벌 규칙).
4. **부분 수정 우선**: 파일 전체 재작성 금지. 특정 라인만 수정.
5. **기존 주석/로직 보존**: 임의 삭제 금지.

## 1. 프로젝트 하드 제약 (ABSOLUTE — 위반 시 검수 반려)

| 항목 | 제약 |
|------|------|
| **SDK 버전** | **2.x 의무** (1.x 빌드는 2026-03-23부터 콘솔 업로드 불가) |
| **렌더링 방식** | **CSR 또는 SSG만 허용**. SSR 절대 금지 (Next.js → `output: 'export'` 필수) |
| **번들 크기** | 압축 해제 기준 **100MB 이하** |
| **외부 코드 실행** | **금지** — `eval`, `new Function`, 동적 `import('http://...')`, 외부 JS 주입 모두 불가 |
| **브라우저 history 조작** | **금지** — `history.pushState/replaceState` 직접 조작 불가 (라우터 추상화만 사용) |
| **테마** | **라이트 모드 필수**. 다크 모드 자동 적용 코드 제거 또는 라이트 강제 |
| **인터랙션 응답** | **2초 이내** 반응 |
| **제스처 줌** | 비활성화 (필수 기능인 경우만 예외 + 사유) |
| **연령** | 만 19세 이상 대상 |
| **최소 OS** | Android 7+, iOS 16+ |
| **`appName`** | 콘솔 등록 후 **수정 불가** — 결정 시 사용자 확인 필수 |

> ⚠️ **AI는 위 11개 항목 중 하나라도 위반하는 코드를 절대 자동 생성·제안하지 않는다.** 위반이 불가피하면 사용자에게 사유를 설명하고 승인받는다.

## 2. 검수 통과 필수 기능 (비게임 체크리스트)

### 2-1. 내비게이션 바 (반려 1순위 항목)
- [ ] **앱인토스 비게임 전용** 내비게이션 바 컴포넌트 사용 (자체 헤더 금지)
- [ ] 좌측 **뒤로가기 버튼**: 모든 화면에서 정상 동작
- [ ] 중앙: **브랜드 로고 + 미니앱 한글 이름**
- [ ] 우측: 미니앱 기능 버튼 **최대 1개**
- [ ] 토스 내비게이션 바 + 자체 뒤로가기 **중복 금지**

### 2-2. 라우팅·이동
- [ ] 미니앱 진입 시 정상 로딩 + 모든 기능이 외부 동일 서비스와 동일 동작
- [ ] 앱 스킴 연결 정상
- [ ] 안드로이드 하드웨어 뒤로가기 정상 동작
- [ ] 화면 가시성(visibility) 변화 시 상태 유지

### 2-3. 보안
- [ ] 외부 코드 실행 경로 없음 (CSP, 동적 코드 로딩 모두 점검)
- [ ] history 직접 조작 없음
- [ ] 민감정보 로컬 저장 시 안전한 방식 사용

### 2-4. 콘텐츠 가이드라인
- [ ] 비속어/과도한 유행어 미포함
- [ ] 불법·선정적 콘텐츠 없음
- [ ] 라이트 모드 화면이 정상 렌더링

### 2-5. 디자인 일관성
- [ ] **토스 디자인 시스템(TDS)** 컴포넌트 우선 사용
- [ ] 미니앱 브랜딩 가이드 준수 (로고·이름·컬러·내비게이션 바·브릿지·탭바)

### 2-6. 선택 기능 사용 시 별도 가이드 충족
다음 기능 사용 시 각 가이드 페이지를 반드시 추가 확인:
- 토스 로그인
- 인앱 결제 (IAP)
- 토스페이
- 인앱 광고
- 공유 리워드

## 3. 단계별 워크플로우

### Phase 0 — 사전 준비 (콘솔 셋업)
**산출물**: 콘솔 워크스페이스 + 등록된 앱 + 결정된 `appName`

1. 토스 비즈니스 회원 가입 (사용자가 직접 — 본인 명의 토스앱 필요)
2. 콘솔 워크스페이스 생성 (사업자당 1개)
3. 멤버 초대 (대표관리자/관리자/구성원 권한 분배)
4. 앱 등록 — **AI는 다음 항목 결정을 사용자와 함께 확정**:
   - 미니앱 한글명, **`appName`(영문, 수정 불가)**
   - 카테고리, 키워드
   - 로고, 썸네일, 스크린샷
   - 고객센터 정보
5. 사업자 등록 여부 확인 → 미보유 시 수익화·토스 로그인 제한 안내

### Phase 1 — 개발 환경 셋업
**산출물**: SDK 2.x 기반 보일러플레이트, 로컬 실행 가능

1. **SDK 선택 의사결정** — 사용자와 합의
   - WebView SDK: 기존 웹앱 자산 재활용·개발 속도 우선
   - React Native SDK: 네이티브 성능·UX 우선
2. 샘플 저장소 클론 또는 참조: https://github.com/toss/apps-in-toss-examples
3. `AppsInToss.registerApp(appName, context)` 등록 — 자동 활성화되는 기능:
   - 파일 기반 라우팅
   - URL 스킴 쿼리 파라미터 처리
   - 뒤로가기 제어
   - 화면 가시성 감지
4. `AppContainer`에서 `InitialProps`(테마, 네트워크, 폰트 크기) 처리 코드 작성
5. **빌드 환경 검증**: CSR/SSG 빌드만 가능하도록 설정 고정
   - Next.js 사용 시 `output: 'export'` + `getServerSideProps` 사용 금지 가드
6. 라이트 모드 강제 (다크 모드 미디어쿼리/시스템 설정 무시)

### Phase 2 — 핵심 기능 개발
**산출물**: 비게임 가이드 충족하는 모든 화면

1. **내비게이션 바**: 비게임 전용 컴포넌트 적용, 우측 버튼 1개 제한 검증
2. **라우팅**: 파일 기반 라우팅 + 뒤로가기 중복 방지
3. **상태 보존**: 화면 가시성 변화 시 데이터 유지
4. **반응 속도**: 모든 인터랙션 2초 이내 응답 (긴 작업은 로딩 UI + 비동기)
5. **TDS 컴포넌트**로 UI 구현 → 브랜드 요소(로고/컬러)는 가이드 범위 내에서 적용

### Phase 3 — 토스 SDK/API 연동 (선택 기능)
사용하는 기능별로 해당 가이드 충족:
- 토스 로그인 → 사업자 필수
- 인앱 결제 / 토스페이 → 결제 흐름·환불·실패 시나리오 구현
- 인앱 광고 → 노출 정책 준수
- 공유 리워드 → 어뷰징 방지

### Phase 4 — 검수 전 자가 점검
**산출물**: §2 체크리스트 모두 ✅, 번들 100MB 이하

1. §2 체크리스트 전수 점검 (AI가 자동으로 수행, 위반 항목 보고)
2. 번들 크기 확인 → 100MB 초과 시 이미지/사운드/영상을 **CDN/외부 스토리지** 분리
3. 샌드박스 환경 테스트 (모든 화면, 모든 흐름)
4. 토스앱 실기기 테스트 (Android 7+, iOS 16+ 최소 1대씩)
5. 라이트 모드 강제 동작 확인
6. 뒤로가기 동작 확인 (하드웨어 키 포함)

### Phase 5 — 검수 요청 및 출시
1. 콘솔에서 빌드 번들 업로드 (한 번에 한 버전)
2. **검토 요청하기** 클릭 → 영업일 기준 최대 3일 소요
3. 반려 시 → 사유 분석 → 수정 → 재요청
4. 승인 후 **출시하기** → **즉시 전체 사용자 노출** (충분한 사전 테스트 후 클릭)

> ⚠️ AI는 사용자의 명시적 지시 없이는 **검토 요청·출시 버튼을 클릭하라고 권하지 않는다.** 이 작업은 사용자가 직접 결정.

### Phase 6 — 출시 후 운영
1. **모니터링**: 오류 로그, API 응답, 사용자 피드백 (도구 사용자와 합의)
2. **새 버전 배포**: 동일 검토 절차 반복
3. **롤백 준비**: 이전 버전으로 복원 가능 — 긴급 시 즉시 롤백
4. **긴급 문의**: 채널톡 (사용자가 직접)

## 4. AI 절대 금지 행동 (Anti-patterns)

| 금지 | 이유 |
|------|------|
| SSR 코드 작성·제안 | 검수 반려 |
| `eval`/`new Function`/외부 JS 동적 로딩 | 검수 반려 |
| `history.pushState/replaceState` 직접 호출 | 검수 반려 |
| 다크 모드 강제 또는 시스템 다크 자동 적용 | 검수 반려 |
| 자체 뒤로가기 버튼 + 토스 내비 동시 노출 | 검수 반려 |
| 우측 내비 버튼 2개 이상 | 검수 반려 |
| 100MB 초과 번들로 업로드 시도 | 업로드 실패 |
| SDK 1.x API 사용 | 빌드/업로드 실패 |
| `appName` 임의 변경 제안 | 콘솔에서 수정 불가 |
| 검수 미통과 항목 남긴 채 "출시하기" 권유 | 사용자 사고 위험 |
| 사용자 동의 없이 콘솔 작업(검토 요청·출시) 실행 | 비가역적 행위 |

## 5. 변경 시 자동 트리거 체크 (코드 수정 시 AI가 매번 셀프 검증)

코드/설정 수정 후 다음을 자동 점검하고 사용자에게 결과 보고:

1. **렌더링 모드**: SSR 코드 추가 안 됨? (`getServerSideProps`, `next/headers` 등)
2. **외부 코드 실행**: 새로 추가된 코드에 동적 코드 로딩 없음?
3. **history**: 라우터 추상화 외에 직접 조작 없음?
4. **테마**: 다크 모드 분기 추가 안 됨?
5. **번들 영향**: 큰 라이브러리/리소스 추가 시 100MB 영향 추정
6. **내비게이션**: 헤더/뒤로가기 관련 변경 시 §2-1 재검증
